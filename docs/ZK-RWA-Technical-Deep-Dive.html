ZK-RWA Allowlist

操作流程 × 痛點解析 × 技術實現完全手冊

第一部分：使用者操作流程（以及為什麼這樣設計）

整個系統有兩種角色：Issuer 發行方 和 Holder 投資者。下面按照實際操作順序，逐步解釋每一步做了什麼、為什麼要做、解決了什麼痛點。

Step 1：Holder 建立身份（/join）

1
操作：投資者打開 /join 頁面，按下「Create identity」。
實際發生的事：瀏覽器在本地隨機產生一組 Semaphore identity（包含 trapdoor 和 nullifier 兩個隨機數），存進 localStorage。然後從這組密鑰計算出一個 commitment（一長串數字）顯示在畫面上。
commitment 是什麼？
commitment = Poseidon 雜湊函數(trapdoor, nullifier)。
它就像你的「會員編號的指紋」— 可以公開交給 issuer，但無法反推出你的密鑰。
就算 issuer 被駭，拿到你的 commitment 也無法冒充你生成 proof。
痛點 → 解法
痛點：傳統 KYC 模式中，投資者必須直接把自己的錢包地址交給 issuer 寫進鏈上名單，導致地址與身份直接綁定且公開。
解法：投資者只交出 commitment（一個無法反推的 hash），而不是錢包地址。地址到後面的 admit 步驟才會出現，而且在鏈上完全無法追溯回這個 commitment。

Step 2：Issuer 加入成員到群組（/issuer → Add）

2
操作：Issuer 在 dashboard 貼上 holder 的 commitment，按「Add to off-chain group」。
實際發生的事：前端呼叫 POST /api/issuer/add-member。伺服器把 commitment 寫進 data/group.json，然後用 Semaphore 的 Group 類別把所有 commitment 建成一棵 Merkle tree，計算出新的 root。
Merkle tree 在這一步做了什麼？
想像把 N 個 commitment 像建金字塔一樣，兩兩配對做 hash，一直往上疊，最後得到一個頂端的值 = Merkle root。
這棵樹有一個超強特性：只需要 root + 一條 path（O(log N) 個 hash），就能證明某個 commitment 確實是樹的一部分。不需要知道完整的樹。
痛點 → 解法
痛點：ERC-3643 需要在鏈上逐一註冊每個投資者地址，每一筆都是公開事件。
解法：名單只存在 issuer 的伺服器上（一個 JSON 檔案），完全不上鏈。後面只會上傳一個 32 bytes 的 root。

Step 3：Issuer 發布 Merkle Root（/issuer → Publish）

3
操作：Issuer 按下「Publish current root on-chain」。
實際發生的事：前端呼叫 POST /api/issuer/publish-root。伺服器用 issuer 的私鑰簽署一筆交易，呼叫鏈上的 ComplianceGate.updateRoot(newRoot)。合約把 root 存入 currentRoot，並加入歷史環形緩衝區。
為什麼需要「發布」這一步？
因為 ZK proof 是「我的 commitment 在某個 root 對應的 Merkle tree 裡面」。
合約需要知道哪些 root 是合法的（由 issuer 發布過的），才能驗證 proof。
如果 root 不在合約裡，proof 就無效 — 防止有人偽造 root。
痛點 → 解法
痛點：ERC-3643 每次加一個投資者都要發一筆鏈上交易（公開的 IdentityRegistered 事件）。
解法：Issuer 可以在 off-chain 一次加入 100 個成員，最後只發布一次 root。一筆交易、一個 hash、零資訊洩露。外部只看到「root 更新了」，完全不知道加了幾個人、是誰。

Step 4：Holder 產生 ZK Proof 並自我 Admit（/admit）

4
操作：投資者打開 /admit，連接錢包，按「Generate proof & admit」。
實際發生的事（分 4 小步）：
步驟 發生什麼 在哪裡執行
4a 前端呼叫 GET /api/group/merkle-path?commitment=...
伺服器查 group.json，找到你的 commitment 在 Merkle tree 中的位置，回傳一條 Merkle path（你的兄弟節點們的 hash）。		 伺服器
4b 瀏覽器拿到 path 後，用 Semaphore SDK 的 generateProof() 開始在 WASM 裡跑 Groth16 proving。
輸入：identity（localStorage 裡的密鑰）、Merkle path、message（你的錢包地址）、scope（token 地址）。
輸出：一個零知識證明（8 個橢圓曲線上的點）+ nullifier + root。
耗時約 3 秒。		 瀏覽器
4c 瀏覽器呼叫 MetaMask 簽署交易，送出 ComplianceGate.admit(token, root, depth, nullifier, points) MetaMask
4d 合約驗證 proof（呼叫 SemaphoreVerifier，用 bn254 pairing 運算）。
通過後：admitted[token][你的地址] = true。
失敗則 revert。		 鏈上
ZK proof 在數學上到底證明了什麼？
一個 Groth16 proof 讓合約確信以下三件事同時為真：
① 我知道一個 commitment C，C 在某個 Merkle root R 對應的樹裡面（= 我在名單上）
② 我用的 message = 我的錢包地址（= 別人不能拿我的 proof 冒充 admit）
③ 我用的 scope = 這個 token 的地址（= 我對 token A 的 admit 不會被偷用在 token B）

但合約完全不知道 C 是什麼。它只看到 root、nullifier、proof 通過了。零知識。
痛點 → 解法
痛點：ERC-3643 中，投資者被 issuer 直接寫進鏈上名單。外部可以看到「0xABC 被加入了 XX token 的白名單」，直接知道是誰。
解法：在我們的系統裡，是投資者自己發起 admit 交易。外部看到的是「0xABC 被 admit 了」，但完全不知道 0xABC 對應群組中的哪一個 commitment。而且群組裡沒 admit 的成員，在鏈上完全不存在。

Step 5：Issuer Mint Token（/issuer → Mint）

5
操作：Issuer 在 dashboard 輸入 holder 的地址和數量，按 Mint。
實際發生的事：伺服器用 issuer 私鑰簽署 PrivateRWA.mint(to, amount)。合約的 _update hook 會檢查 to 是否 admitted。如果 admitted → mint 成功。如果沒有 → revert NotAdmitted。
痛點 → 解法
痛點：普通 ERC-20 的 mint 不做任何合規檢查，任何地址都能收到 token。
解法：我們的 token 在最底層（_update hook）就強制檢查。就算是 issuer 也不能 mint 給沒 admit 的地址。合規是在合約層面硬編碼的，不靠人工判斷。

Step 6：Holder 轉帳（/wallet → Transfer）

6
操作：Holder 在 wallet 頁面輸入對方地址和數量，按 Transfer。
實際發生的事：MetaMask 簽署 PrivateRWA.transfer(to, amount)。合約 _update 檢查 from 和 to 都 admitted → 通過則轉帳成功，不通過則 revert NotAdmitted(address)。
你之前那筆「失敗的」transfer 就是這個機制在運作！
你轉帳給 0x26c0…，但那個地址沒做過 admit。合約直接 revert 了。
這不是 bug — 這是功能。這正是我們要展示的核心 demo 環節：
「合規不是靠 off-chain 監控，是合約本身在擋。沒 admit 過，一毛錢都轉不出去。」
痛點 → 解法（總結）
痛點：ERC-3643 在鏈上做合規，但名單公開。不做 ERC-3643 則無法合規。
解法：我們做到了跟 ERC-3643 同樣的合規效果（transfer 被合約攔截），但白名單是隱私的。
機構得到他們需要的一切：合規轉帳 + 投資人名單保密。

ERC-3643（T-REX）是目前 RWA token 的合規標準，但它的身份註冊表（Identity Registry）完全公開。任何人都能列舉出所有通過 KYC 的投資者地址。這對機構投資者來說是不可接受的隱私洩露。

3.2 ComplianceGate.sol — 核心合約

contracts/src/ComplianceGate.sol（約 110 行）

這個合約是整個系統的心臟。它負責三件事：

功能 A：updateRoot — 儲存 Merkle root

function updateRoot(uint256 newRoot) external onlyIssuer {
    currentRoot = newRoot;
    rootHistory[rootHistoryHead] = newRoot;                // 存進環形緩衝區
    rootHistoryHead = (rootHistoryHead + 1) % 16;          // 最多保留 16 個
    knownRoot[newRoot] = true;                             // 標記為已知 root
    emit RootUpdated(newRoot);
}
為什麼用環形緩衝區保留 16 個歷史 root？
因為 holder 可能在 root_v5 的時候產生了 proof，但 issuer 在他送出 tx 之前又更新了 root 到 v6。
如果只接受最新 root，holder 的 proof 就會失敗。保留 16 個歷史 root 給了一個寬裕的時間窗口。
knownRoot[root] mapping 讓 admit 函數可以快速查詢某個 root 是否合法。

功能 B：admit — ZK 驗證 + 准入記錄

function admit(
    address token,      // 要 admit 到哪個 RWA token
    uint256 root,       // proof 是基於哪個 Merkle root 生成的
    uint256 depth,      // Merkle tree 深度
    uint256 nullifier,  // 防重複使用的唯一值
    uint256[8] points   // Groth16 proof（8 個橢圓曲線點）
) external {

    // 1. 基礎檢查
    if (!registeredToken[token]) revert TokenNotRegistered();
    if (!knownRoot[root])        revert UnknownRoot();
    if (usedNullifier[nullifier]) revert NullifierReused();

    // 2. 計算 message hash 和 scope hash
    uint256 messageHash = _hashToField(uint256(uint160(msg.sender)));
    uint256 scopeHash   = _hashToField(uint256(uint160(token)));

    // 3. 呼叫 Semaphore Verifier 做 Groth16 pairing 驗證
    bool ok = verifier.verifyProof(
        [points[0], points[1]],                          // pA
        [[points[2], points[3]], [points[4], points[5]]], // pB
        [points[6], points[7]],                          // pC
        [root, nullifier, messageHash, scopeHash],          // 公開信號
        depth
    );
    if (!ok) revert InvalidProof();

    // 4. 記錄准入
    usedNullifier[nullifier] = true;
    admitted[token][msg.sender] = true;
    emit Admitted(token, msg.sender, nullifier);
}
逐行解析重點：

messageHash = _hashToField(msg.sender)
合約自己算 msg.sender 的 hash，而不是從 proof 裡拿。這確保 proof 是「為我的地址」生成的。假如 Alice 的 proof 被 Bob 截獲，Bob 不能用這個 proof 去 admit — 因為合約會用 Bob 的地址算 hash，跟 Alice 的 proof 裡的 hash 不同，驗證就會失敗。

scopeHash = _hashToField(token)
scope 綁定到 token 地址。同一個 identity 可以 admit 到 token A 和 token B，因為 scope 不同，nullifier 也不同。

_hashToField：keccak256(value) >> 8
把 256-bit 值縮到 ~248 bit，讓它落在 BN254 曲線的標量場（scalar field）內。這跟 Semaphore 合約用的是同一個函數，確保鏈上鏈下的 hash 一致。

verifier.verifyProof 內部
呼叫 EIP-196/197 的 bn254 precompiles（地址 0x06、0x07、0x08），執行橢圓曲線配對運算。數學上驗證 Groth16 零知識證明的正確性。約消耗 300K-500K gas。

功能 C：isAdmitted — 查詢某地址是否已准入

function isAdmitted(address token, address holder) external view returns (bool) {
    return admitted[token][holder];
}

就是一個 mapping lookup。被 PrivateRWA 的 _update 每次 transfer 都會呼叫。gas 極低（~2600 gas 的 SLOAD）。

3.3 PrivateRWA.sol — 合規 ERC-20

contracts/src/PrivateRWA.sol（約 45 行）

contract PrivateRWA is ERC20 {
    ComplianceGate public immutable gate;

    function _update(address from, address to, uint256 value)
        internal override
    {
        // mint 時 from == address(0)，跳過 sender 檢查
        if (from != address(0) && !gate.isAdmitted(address(this), from))
            revert NotAdmitted(from);

        // burn 時 to == address(0)，跳過 recipient 檢查
        if (to != address(0) && !gate.isAdmitted(address(this), to))
            revert NotAdmitted(to);

        super._update(from, to, value);
    }
}
為什麼用 _update 而不是 _beforeTokenTransfer？
OpenZeppelin v5 把所有 token 狀態變更統一收歸到 _update。這個 hook 涵蓋 mint、burn、transfer 三種操作。 在這裡做檢查，代表無論什麼途徑移動 token，都必須通過合規檢查。沒有任何繞過的方式。

3.4 identity.ts — 瀏覽器端身份管理

lib/semaphore/identity.ts

import { Identity } from "@semaphore-protocol/identity";

export function createIdentity(): Identity {
    const id = new Identity();               // 隨機產生 trapdoor + nullifier
    localStorage.setItem(KEY, id.export()); // 序列化存入 localStorage
    return id;
}

export function loadIdentity(): Identity | null {
    const raw = localStorage.getItem(KEY);
    if (!raw) return null;
    return Identity.import(raw);            // 反序列化
}
Identity 裡面有什麼？
new Identity() 會產生兩個隨機的 256-bit 數字：
• trapdoor：用於計算 commitment = Poseidon(trapdoor, nullifier)
• nullifier：用於產生 proof 時的 nullifier hash（防重複）

id.commitment = Poseidon(trapdoor, nullifier) — 這就是你複製給 issuer 的那串數字。
只有同時知道 trapdoor 和 nullifier 才能生成有效的 ZK proof。

3.6 proof.ts — 瀏覽器端 ZK Proof 生成

lib/semaphore/proof.ts

export async function generateAdmissionProof(
    identity: Identity,       // 從 localStorage 載入的密鑰
    merkleProofInput,         // 從 /api/group/merkle-path 拿到的 path
    holderAddress,            // MetaMask 連接的錢包地址
    tokenAddress              // 要 admit 的 token 合約地址
): Promise<AdmissionProof> {

    const message = BigInt(holderAddress);  // 綁定到錢包
    const scope   = BigInt(tokenAddress);   // 綁定到 token

    // Semaphore SDK 內部：
    // 1. 載入 .wasm 和 .zkey artifacts（Groth16 電路編譯後的檔案）
    // 2. 用 snarkjs 在 WASM 裡執行 Groth16 proving
    // 3. 約 3 秒完成
    const proof = await generateProof(identity, merkleProof, message, scope);

    return {
        root: proof.merkleTreeRoot,
        depth: proof.merkleTreeDepth,
        nullifier: proof.nullifier,
        points: proof.points,       // 8 個 BN254 曲線上的點
    };
}
proof.points 是什麼？
Groth16 proof 由三個橢圓曲線點組成：pA (2 coords), pB (4 coords), pC (2 coords) = 8 個數字。
合約把它們傳給 SemaphoreVerifier，verifier 用 EIP-196/197 precompile 做配對運算（bilinear pairing）。
如果配對結果正確 → proof 有效 → 返回 true。數學上偽造這個 proof 等同於破解離散對數問題。

1. 部署 SemaphoreVerifier      ← Semaphore 官方的 Groth16 verifier（bn254 pairing）

3. 部署 PrivateRWA("HKGB30", gate, issuer) ← 指向 gate + issuer
4. 呼叫 gate.registerToken(rwa)    ← 把 token 註冊到 gate

現在你應該完全理解了：

① 痛點：ERC-3643 把投資人名單公開在鏈上，機構不敢用。
② 解法：名單存 off-chain，鏈上只有 Merkle root hash。投資者用 ZK proof 自己 admit。
③ 合規：transfer hook 在合約層面強制執行，沒 admit 就轉不了帳。
④ 隱私：外部能看到誰 admitted，但不知道群組裡還有誰、也不知道地址對應哪個 commitment。
⑤ 技術：零自訂 ZK 電路，用 Semaphore v4 當黑盒。兩個合約加起來 < 160 行。
